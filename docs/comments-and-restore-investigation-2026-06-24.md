# 评论屏蔽与恢复调查

日期：2026-06-24  
对应原始记录：`docs/investigation-comments-and-restore-raw-2026-06-24.md`

## 目的

这次调查的目标不是立刻改行为，而是先把下面三件事分清楚：

1. 视频页评论区的“单条评论屏蔽/恢复”是怎么工作的
2. “因为评论命中而屏蔽整条视频”的链路是怎么工作的
3. 视频卡片 `hide / overlay` 双路径目前是实际 bug，还是工程实现上不够稳

用户当前要求是：

- 保留两条路径
- 优先梳理评论相关逻辑
- 视频端如果没有明确问题，先不要把工程风险误判成现网 bug

---

## 结论摘要

### 1. 评论相关其实有两条完全不同的链路

第一条是**评论区 DOM 屏蔽**：

- 作用对象是视频页里已经渲染出来的单条评论
- 根据评论内容、评论用户、带图评论等规则进行折叠
- 不主动请求评论 API

第二条是**评论 API 驱动的视频屏蔽**：

- 作用对象是整条视频卡片
- 根据“精选评论”和“置顶评论”元信息决定视频是否 `blockedTarget`
- 依赖评论 API

这两条线不能混在一起讨论，否则会把“评论恢复”问题和“视频再次命中评论规则”问题混成一个问题。

### 2. 评论区 DOM 的屏蔽/恢复逻辑目前没有看到明确硬 bug

从代码结构上看，评论区这条线是闭环的：

- 命中规则时：隐藏评论、记录原始显示状态、插入占位条
- 手动显示时：恢复评论显示、打 bypass 标记、保留占位条
- 不再命中或总开关关闭时：移除占位条、恢复评论、清理状态

目前没有看到明显的“恢复漏做了某一步”这种硬错误。

### 3. 真正更值得关注的是“时序复杂度”，不是恢复函数本体

评论区这条线同时受到下面几类触发影响：

- 页面级 MutationObserver
- 评论区 Shadow DOM 观察器
- 评论 settling retry
- quick block 后的手动 refresh

所以如果用户体感上觉得“评论恢复不稳”“看起来像没恢复干净”，更应该优先排查触发时序和 DOM 重建，而不是先怀疑恢复函数本体。

### 4. 视频卡片 `hide / overlay` 双路径确实存在工程上的不够稳，但目前不建议直接当 bug 处理

视频卡片渲染和恢复逻辑仍然是按“当前模式开关”分叉处理，而不是按“元素之前到底是被 hide 还是被 overlay”恢复。这个实现方式从工程稳健性上看有风险。

但在当前前提下：

- 用户明确说视频端实测没有看到问题
- 两条路径还要保留

所以更合适的定性是：

> 当前是一个值得记录的工程风险点，不是这次要优先修的现网 bug。

## 范围拆分

## A. 评论区 DOM 屏蔽与恢复

对应文件：

- `src/features/comment-filter.js`
- `src/platform/dom-adapter.js`
- `src/platform/renderer.js`
- `src/actions/comment-quick-block.js`
- `src/orchestration/pipeline.js`

### 行为模型

这条链路的流程是：

1. 在视频页识别评论元素
2. 读取评论文本、用户、是否带图
3. 命中规则则隐藏评论本体
4. 在原位置插一个占位条
5. 用户可以临时“显示”这条评论
6. 规则不命中或总开关关闭时恢复评论本体

### 代码判断

`commentFilterFeature` 的职责很明确：

- 只处理视频页评论区
- 只读取页面已渲染评论
- 不主动请求评论 API

渲染器中的评论逻辑也比较完整：

- `blockCommentElement()` 负责隐藏评论并记录原始显示状态
- `revealCommentElement()` 负责临时显示并保留占位提示
- `restoreCommentElement()` 负责彻底恢复评论和清理状态

我的判断是：

- 这套设计在职责边界上是成立的
- 当前没有看到明显的状态清理遗漏
- 如果后续要修评论“恢复不稳”，大概率不是从 `restoreCommentElement()` 开刀，而是从触发时序排查

### 主要风险

这条线最大的复杂度不在渲染，而在“谁在什么时候触发 refresh”。

目前评论区会被多层机制驱动：

- `observeCommentChanges()` 监听评论 DOM / Shadow DOM 变化
- 评论数量变化后会触发 settling retry
- quick block 操作也会触发 refresh

这意味着：

- 同一条评论可能在短时间内被重复评估
- B 站如果重建了评论 DOM 节点，旧节点上的本地状态会自然丢失，再由新一轮评估重建
- 用户感知到的“恢复异常”有可能只是重跑时序和页面重渲染叠加造成的视觉现象

## B. 评论 API 驱动的视频屏蔽

对应文件：

- `src/features/comments.js`
- `src/platform/api-client.js`
- `src/state/video-store.js`
- `src/platform/api-health.js`

### 行为模型

这条链路的流程是：

1. 发现相关评论规则已开启
2. 根据视频 BV 拉取评论元信息
3. 写入 `filteredComments` / `topComment`
4. 在 `videoStore` 中应用评论相关视频规则
5. 如果命中，则把视频标记为 `blockedTarget`

### 代码判断

这里的实现比表面上更完整：

- 有 `pending / ready / empty / unavailable` 状态机
- 有主接口与 legacy fallback
- 有错峰节流
- 有“已 blocked 是否继续请求 API”的策略控制

我的判断是：

- 这条链不是脆弱拼接，而是相对成形的子系统
- 如果以后用户说“评论规则相关的视频恢复不对”，先要确认他说的是：
  - 评论区某条评论恢复不对
  - 还是整条视频因为评论元信息又被重新命中

这两种情况的排查方向完全不同。

### 需要特别说明的点

`videoStore.resetBlockEvaluation()` 和 `resetAllBlockEvaluations()` 只会清：

- `blockedTarget`
- `triggeredBlockedRules`
- `blockedReasons`

不会清：

- `filteredComments`
- `topComment`
- `apiDataStates`

这意味着：

> 重新评估只是在清“判定结果”，不是在抹掉评论 API 已经拿到的数据。

所以如果评论 API 数据还在，重新评估时再次命中是合理行为，不应误判为“恢复失败”。

## C. 视频卡片 `hide / overlay` 双路径

对应文件：

- `src/platform/renderer.js`
- `src/platform/userscript-menu.js`

### 当前事实

这条线确实有两种模式：

1. `hideVideoMode_Switch = true`
   - 隐藏视频元素
2. `hideVideoMode_Switch = false`
   - 给视频加 overlay

菜单中也明确保留了这个开关。

### 我的判断

从工程角度看，这里的实现有一个不够理想的点：

- 恢复行为会参考“当前模式开关”
- 而不是参考“这个元素之前具体经历了哪种屏蔽方式”

这种实现方式会让代码阅读成本更高，也会让后续维护者更难确信它在所有时序下都安全。

但我现在不建议直接改，因为：

- 用户当前没有确认视频端存在真实问题
- 两条模式路径需要保留
- 现在贸然重构这块，收益不一定高于风险

更合适的做法是：

- 先把它记为工程层风险点
- 等评论主线梳理稳定后，再决定是否单独收这个债

---

## 这次调查后的整体判断

### 可以确认的事实

1. 评论区 DOM 屏蔽 和 评论 API 驱动视频屏蔽 是两条独立链路
2. 评论区恢复逻辑本体当前看是完整的
3. 视频评论相关 API 逻辑并不粗糙，已经有状态、节流和 fallback
4. 视频卡片的双路径设计需要保留

### 我更倾向的判断

1. 当前最像“问题来源”的不是恢复函数本身，而是评论区多重触发带来的时序复杂度
2. 视频端 `hide / overlay` 更像“工程稳健性一般”，不是当前已经被证实的 bug
3. 如果用户未来反馈“评论恢复有问题”，优先要拿浏览器侧复现和触发顺序证据，而不是先动渲染器

---

## 建议

## 1. 先不要改视频卡片双路径行为

原因：

- 需求上明确要保留两条路径
- 现阶段没有复现到视频端真实错误
- 直接改会把“工程洁癖”误当成“现网修复”

建议动作：

- 暂时不改行为
- 只在文档里保留这个风险点

## 2. 下一步优先补评论区的浏览器实测证据

建议重点观察这些场景：

1. 首次进入视频页，评论区渐进加载
2. 下拉加载更多评论
3. 点击占位条中的“显示 / 重新隐藏”
4. 使用 quick block 添加规则
5. 关闭脚本总开关

目标不是复述代码，而是确认：

- 会不会出现重复 refresh
- 同一条评论是否被 B 站替换成新的 DOM 节点
- bypass 状态在页面局部重渲染后是否符合预期

## 3. 给评论区链路补更强的调试可见性

如果要继续维护这块，我建议加最小量调试能力，而不是先改逻辑：

- 记录评论 refresh 的触发来源
- 记录评论数量变化与 retry 次数
- 记录某条评论是“首次 block / 重复 block / reveal / restore”

这样一旦有“恢复异常”反馈，就能更快判断是逻辑问题还是时序问题。

## 4. 适度拆分评论规则评估和 quick block 追踪

当前 `commentFilterFeature` 同时承担：

- 评论规则评估
- 评论变化监听
- quick block 入口挂载

这在功能上可行，但会让后续排查复杂。

我的建议不是马上重构，而是先在设计上把它们区分成两个概念：

1. 评论规则处理
2. 评论现场交互增强

哪怕短期内不拆文件，至少也建议在文档和注释层先把这两层职责说清楚。

## 5. 为评论恢复补最小测试矩阵

即便暂时不写自动化，也建议保留一个手工验证矩阵：

- 规则命中 -> 评论被隐藏
- 规则移除 -> 评论恢复
- 点击“显示” -> 评论出现，占位条保留
- 点击“重新隐藏” -> 评论再次隐藏
- 关闭脚本 -> 评论恢复且占位条消失

这组验证项比继续泛泛地讨论“恢复有没有问题”更有价值。

当前已补第一版自动化入口：

```powershell
npm.cmd run build
npm.cmd run pw:comment-timing -- --video https://www.bilibili.com/video/BV1Vk7M6tEgx/
```

它会连接 `127.0.0.1:9223` 上的专用 Chrome，会把当前 `dist/bilibili_blocked_videos_by_tags.user.js` 注入页面，选取首条已渲染评论的一小段文本作为屏蔽关键词，然后验证：

- 命中规则后评论隐藏并出现占位条
- 点击“显示”后评论恢复且带 bypass
- 触发一次 refresh 后仍保持显示
- 点击“重新隐藏”后评论再次隐藏
- 点击浮窗总开关关闭脚本后评论恢复、占位条消失

运行结果会写到 `artifacts/playwright/comment-timing/<run-id>/result.json` 和 `events.jsonl`，用于后续比对时序变化。

---

## 建议的后续顺序

1. 保留当前双路径，不动视频卡片模式
2. 先做评论区浏览器侧复现与触发时序确认
3. 视证据决定是否需要：
   - 补日志
   - 补注释
   - 轻量拆职责
4. 如果后续视频端确实出现模式切换或恢复异常，再单独开一个视频卡片渲染专题处理

# 技术报告

> 最后更新：2026-06-26  
> 范围：当前重构版 userscript 的实现结构、能力边界、UI 收口与验证方式。

## 项目形态

本项目是 Bilibili 视频屏蔽 userscript 的独立维护重构版。源码以 ESM 拆分在 `src/`，通过 `scripts/build-userjs.mjs` 打包为 `dist/bilibili_blocked_videos_by_tags.user.js`，安装目标是 Tampermonkey / 篡改猴。

配置仍使用 `GM_blockedParameter`，兼容原项目导出的 JSON 配置。运行时缓存只保留在当前页面生命周期内，主要以 BV 号、UP UID 等作为索引。

## 核心架构

| 模块 | 职责 |
|------|------|
| `src/main.js` | 入口组装、菜单注册、页面观察器启动 |
| `src/orchestration/pipeline.js` | 调度页面规则、视频规则、后处理与刷新 |
| `src/features/` | 按功能拆分的规则入口 |
| `src/platform/dom-adapter.js` | 隔离 B 站 DOM 读取、卡片识别和评论读取 |
| `src/platform/api-client.js` | 额外 API 数据拉取、节流与状态记录 |
| `src/platform/renderer.js` | 视频叠加层、隐藏模式、评论遮罩渲染 |
| `src/settings/` | 默认配置、兼容迁移、配置变更 |
| `src/state/` | 视频运行时状态、UP 屏蔽统计 |
| `src/ui/` / `src/actions/` | 浮动入口、统计面板、快速屏蔽、追溯面板 |

## 数据来源与能力边界

规则分为 DOM 能力和 API 能力：

| 类型 | 代表能力 | 风险 |
|------|----------|------|
| DOM | 标题、UP 名、部分 UID、已渲染评论、热搜、非视频元素清理 | 主要受 B 站改版影响 |
| API | 标签、分区、统计数据、UP 资料、置顶/精选评论 | 受接口变更、限频、风控影响 |
| 混合 | UP 规则、白名单、播放页推荐 | 优先 DOM，缺数据时再依赖 API |

API 失败时，本项目的原则是“不误杀”：缺失或不可用的数据不视为命中。能力边界与 API 透明化细节见 [`capability-transparency.md`](capability-transparency.md)。

## UI 收口

当前 UI 做了最后一轮视觉统一：

- 浮动入口、设置面板、快速屏蔽、评论快速屏蔽、追溯面板、统计面板统一为深色工作面板。
- 主操作使用青色强调，危险操作使用红色，统计/阈值类信息使用琥珀色或浅青色。
- 小按钮图标化，包括关闭、设置、保存、导入、导出、读取、统计、JSON、屏蔽、解封、删除等。
- 保留核心文字按钮，避免影响可读性和现有操作习惯。

图标由 `src/ui/icons.js` 统一创建，采用内联 SVG，颜色跟随 `currentColor`，不引入外部依赖。

## 性能结论

轻量边界压测命令：

```bash
npm run perf:boundary
```

当前测试固定 `1000 / 2000 / 5000` 三档，含义如下：

| 档位 | 定位 | 工程结论 |
|------|------|----------|
| 1000 条/名单 | 稳定使用档 | 视频规则整体压力很低；评论普通关键词仍可接受；评论正则和评论用户名单已经明显比其他规则重。 |
| 2000 条/名单 | 压力目标档 | 视频普通规则仍为毫秒级；视频正则和评论普通关键词属于可接受压力；评论正则、评论用户名单会产生明显主线程开销。 |
| 5000 条/名单 | 探索上限档 | 存储仍可承受，但只建议作为视频普通名单上限；正则和评论相关名单不应按 5000 作为默认承诺。 |

关键结论：

- 存储不是主要瓶颈：5000 条/名单时，8 个主要名单共 40000 条，配置约 0.58 MiB，保存/读取 p95 仍在十几毫秒量级。
- 视频普通规则压力较低：标题、UP UID、白名单、标签普通匹配在 2000 档仍为毫秒级，5000 档也未成为主要风险。
- 正则规则需要单独看待：视频标题正则在 2000 档可接受，5000 档已进入明显卡顿风险区。
- 评论规则是主要风险：评论正则和评论用户名单在 2000 档会产生明显主线程开销，5000 档不建议作为可用规模。
- 核心路径未访问页面 `localStorage` / `sessionStorage`。

详细数据见 [`performance-boundary-report-2026-06-26.md`](performance-boundary-report-2026-06-26.md)。

## 建议上限

| 名单类型 | 建议上限 |
|----------|----------|
| 视频普通名单 | 5000 条以内 |
| 视频正则名单 | 2000 条以内 |
| 评论普通关键词 | 2000 条以内 |
| 评论正则关键词 | 500-1000 条以内 |
| 评论用户名单 | 500-1000 条以内 |

超过上述规模时，脚本不一定保存失败，但滚动、评论加载和重复刷新可能出现明显卡顿。

## 验证方式

常规验证：

```bash
npm run check
```

单独测试：

```bash
npm test
npm run perf:boundary
```

浏览器烟测可使用现有 Playwright 脚本：

```bash
npm run pw:smoke
npm run pw:video-card-timing
npm run pw:comment-timing
```

## 后续维护重点

1. B 站 DOM 改版时，优先维护 `dom-adapter.js`。
2. API 失效时，优先补透明提示和 DOM 降级，避免 silent fail。
3. 评论用户名单和评论正则如需支持更大规模，应考虑预索引、正则缓存或更明确的 UI 限制。
4. `dist/` 为构建产物，发布前以 `npm run build` 生成。

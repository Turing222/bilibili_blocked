# 性能边界报告

> 日期：2026-06-26  
> 命令：`npm run perf:boundary`  
> 目的：用轻量压测对比理论检查次数和本地核心实现耗时，判断名单规模建议上限。

## 结论总览

| 档位 | 结论 | 建议口径 |
|------|------|----------|
| 1000 条/名单 | 稳定档。视频规则、评论普通关键词整体可接受；评论正则和评论用户名单已明显更重。 | 可作为多数名单的舒适规模，评论正则/用户名单仍建议谨慎。 |
| 2000 条/名单 | 压力目标档。视频普通规则仍轻；视频正则和评论普通关键词基本可接受；评论正则、评论用户名单可能明显拖慢评论区。 | 可作为视频正则、评论普通关键词的建议上限；不建议作为评论正则/用户名单的常规上限。 |
| 5000 条/名单 | 探索上限档。存储仍能承受，视频普通规则仍可用；正则和评论相关名单风险明显。 | 只建议承诺给视频普通名单，不建议承诺给正则或评论名单。 |

因此最终用户口径建议为：

| 类型 | 建议 |
|------|------|
| 视频普通名单（标题、UP UID、标签、白名单） | 5000 条以内通常可接受 |
| 视频正则名单 | 建议 2000 条以内 |
| 评论普通关键词 | 建议 2000 条以内 |
| 评论正则关键词 | 建议 500-1000 条以内 |
| 评论用户名单 | 建议 500-1000 条以内，2000 条以上可能明显卡顿 |

这些数字是本地核心路径轻量压测结论，不是浏览器真实页面的硬限制。真实体验还会受页面评论量、浏览器性能、B 站 DOM 改版和其他 userscript 影响。

## 测试设计

固定三档：

| 档位 | 含义 |
|------|------|
| 1000 条/名单 | 期望稳定使用 |
| 2000 条/名单 | 压力目标 |
| 5000 条/名单 | 探索上限 |

存储测试覆盖 8 个主要名单字段：

- `blockedTitle_Array`
- `blockedUpUid_Array`
- `blockedUpNameKeyword_Array`
- `blockedTag_Array`
- `blockedCommentText_Array`
- `blockedCommentUser_Array`
- `whitelistUpUid_Array`
- `whitelistBv_Array`

规则测试使用：

- 100 个视频卡片
- 每个视频 6 个标签
- 500 条评论
- 最坏情况为“不命中”，即需要扫完整个名单

## 存储结果

| 档位 | 总条目 | 配置体积 | 保存 p95 | 读取 p95 | 回读 |
|------|--------|----------|----------|----------|------|
| 1000 | 8000 | 0.12 MiB | 1.61ms | 1.98ms | PASS |
| 2000 | 16000 | 0.23 MiB | 2.78ms | 5.41ms | PASS |
| 5000 | 40000 | 0.58 MiB | 8.15ms | 18.78ms | PASS |

结论：当前结构下，普通短名单的 GM 配置存储不是瓶颈。即使 5000 条/名单，配置体积也仍低于 1 MiB。

## Web Storage 检查

压测脚本安装了 tripwire：

| localStorage 调用 | sessionStorage 调用 | 结果 |
|-------------------|---------------------|------|
| 0 | 0 | PASS |

结论：核心配置路径不访问页面自身 Web Storage。脚本主要依赖 Tampermonkey 的 GM 存储。

## 体感判断

浏览器主线程中可按以下经验判断：

| 单次耗时 | 体感 |
|----------|------|
| < 50ms | 基本无感 |
| 50-100ms | 可能轻微卡顿 |
| 100-300ms | 明显卡顿 |
| > 300ms | 明显影响滚动或交互 |
| > 1000ms | 接近卡死感 |

因此：

- 视频普通规则在 2000-5000 条范围内仍较安全。
- 视频正则在 2000 条内可接受，5000 条开始有明显风险。
- 评论普通关键词 2000 条内可接受，5000 条开始有明显风险。
- 评论正则和评论用户名单在 2000 条时已经可能明显影响评论区加载。

## 建议口径

对用户说明时建议写成：

| 类型 | 建议 |
|------|------|
| 视频普通名单 | 5000 条以内 |
| 视频正则名单 | 2000 条以内 |
| 评论普通关键词 | 2000 条以内 |
| 评论正则关键词 | 500-1000 条以内 |
| 评论用户名单 | 500-1000 条以内，2000 条以上可能明显卡顿 |

## 后续优化方向

如果未来要支持更大规模评论名单，优先级如下：

1. 评论用户名单预拆为 UID Set、name Set，避免每条评论扫完整数组。
2. 正则规则保存时预编译或运行时缓存，避免匹配时重复 `new RegExp(pattern)`。
3. 评论文本普通关键词按长度、首字符或分词做轻量索引。
4. UI 中对评论正则和评论用户名单给出数量提示或软上限。

# 能力边界与 API 透明化方案（讨论稿）

> 状态：基础透明化已实现（能力元信息、API 统一记录、单视频数据状态）；熔断与设置 UI 展示待实现。  
> 用途：自用维护 + 技术论坛讨论。  
> 最后更新：2026-06-24

## 背景

本脚本的屏蔽规则分两类数据来源：

1. **页面已有（DOM）**：从 B 站已渲染的卡片/评论区读取标题、UP、热搜等。
2. **额外请求（API）**：向 `api.bilibili.com` 拉标签、分区、统计数据、评论元信息等。

B 站 Web 端接口无公开 SLA，近年持续收紧（WBI 签名、限频、风控）。原版作者也在菜单里提示过评论 API「极易被拒绝」。  
因此：**不是整个脚本会废掉，而是依赖 API 的规则会越来越不稳定。**

当前代码里，API 失败大多只打控制台日志，用户容易以为「规则没配好」或「脚本坏了」。  
本方案的目标是把**能力边界讲清楚**，并把**失败情况显式暴露**，而不是隐藏或过度承诺。

---

## 方案目标

| 目标 | 说明 |
|------|------|
| 区分数据来源 | 每个（或每组）功能标明 DOM / API / 混合 |
| 标明风险等级 | 低 / 中 / 高，便于决定维护优先级 |
| 失败上移 | API 失败在设置或调试区可见，而不是 silent fail |
| 保持简单 | 文档 + 少量 UI 提示；不做复杂监控平台 |

**不做的事（第一版）：**

- 不做全站 API 代理、不做与 B 站风控硬刚
- 不在每次失败时弹窗打断浏览
- 不对外宣称「B 站官方 API 状态」（只记录脚本侧观测）

---

## 功能分类（当前实现）

### 一、页面已有（DOM）— 风险：低

主要风险来自 B 站改版导致 selector 失效，与 API 风控无关。

| 功能 | 说明 |
|------|------|
| 按标题屏蔽 | 读卡片 `title` |
| 按 UP 名 / UID 屏蔽 | 读卡片内 space 链接（多数页面足够） |
| 隐藏热搜 / 按关键字屏蔽热搜 | 读热搜 DOM |
| 隐藏首页等非视频元素 | 读页面 DOM 隐藏广告/推广 |
| 视频页按评论内容屏蔽 | 读已渲染评论（Shadow DOM），**不主动调评论 API** |
| 叠加层 / 隐藏模式 / 导入导出 | 与数据获取无关 |

**维护策略：** 长期核心，优先保证。

---

### 二、额外 API — 风险：中

接口集中在 `src/platform/api-client.js`，目前统一经 API 适配层记录状态，无 WBI 签名。

| 功能 | 依赖接口 | 默认是否开启 |
|------|----------|--------------|
| 按标签 / 双重标签屏蔽 | `/x/web-interface/view/detail/tag` | **开** |
| 按视频分区屏蔽 | `/x/web-interface/view`（必要时 `/dynamic/region`） | **开** |
| 时长 / 播放量 / 赞币率 / 收藏投币比 | `/x/web-interface/view` | 关 |
| 竖屏 / 充电专属 | `/x/web-interface/view` | 关 |
| UP 等级 / 粉丝 / 简介 | `/x/web-interface/view` + `/x/web-interface/card` | 关 |

**已知情况：**

- 在浏览器页面内调用，比站外爬虫稳，但仍可能遇到 `code: -403`、空 `data`、限频。
- 2023 年起部分 Web 接口纳入 WBI；本项目尚未实现签名。
- **默认配置中，标签 + 分区开关为开**；当用户配置了对应规则项后会调 API，用户即使不开更高级的统计/评论规则也会受影响。

**维护策略：** 按需做 WBI、DOM 降级；透明标注，不当作「永久可靠能力」。

---

### 三、额外 API — 风险：高

| 功能 | 依赖接口 | 默认是否开启 |
|------|----------|--------------|
| 屏蔽精选评论的视频 | `/x/v2/reply/main` → 回退 `/x/v2/reply` | 关 |
| 按置顶评论屏蔽 | 同上 | 关 |

**已知情况：**

- 原版即标注为频率敏感；列表页批量请求极易被拒。
- 社区文档显示评论主路径已迁移至 WBI 版（如 `/x/v2/reply/wbi/main`），当前实现未签名。
- 历史上是最先出现「开了但不生效」的功能组。

**维护策略：** 设置里标「实验 / 高失败率」；默认保持关闭；论坛讨论中如实说明限制。

---

### 四、混合

| 功能 | 行为 |
|------|------|
| 按 UP 名 / UID 屏蔽 | 优先 DOM；部分页面缺 UID 时会补调 `view` |
| 白名单 | 规则本身是 DOM/配置；不额外增加 API |

## 「失败上移」怎么做（拟）

### 原则

1. **API 层统一记录**：endpoint、HTTP 状态、`json.code`、时间、连续失败次数。（已在 `src/platform/api-health.js` + `src/platform/api-client.js` 落地）
2. **按能力聚合展示**：例如「标签 API：最近 10 次，7 成功 / 3 失败（-403）」。
3. **规则语义写清**：API 无数据或不可用时，当前行为是**该规则未判定（不视为命中）**，避免用户以为「已判定未命中」或「误杀」。
4. **分级暴露**：
   - 设置 UI：功能旁短标签 + 最近一次失败摘要（主要）
   - 调试开关：详细 log（自用）
   - 不做：每次失败弹 toast（太吵）

### 状态示意（UI 文案草案）

- `正常` — 近期请求成功
- `不稳定` — 间歇失败
- `不可用` — 连续失败超过阈值（如 5 次 `-403`）

---

## 为什么选透明化，而不是 silent 修补

1. **使用场景**：自用 + 技术论坛分析；读者需要知道「靠什么数据、何时不可信」。
2. **维护成本**：B 站改 DOM + 收紧 API 是持续成本；透明化后不必为 API 失效背「脚本 bug」的锅。
3. **产品取舍**：把重心放在 DOM 核心能力；API 高级规则如实标注为「可能失效」。
4. **与原版一致、但更系统**：原版已有 `title` 提示；本方案是把它结构化，而不是推翻重来。

## 实施范围（建议分步落地）

| 阶段 | 内容 |
|------|------|
| 文档 | 本文 + README 链出一行 |
| 能力元信息 | `src/capabilities/registry.js` 维护数据来源、风险、端点、失败语义 |
| API 层 | 统一错误记录与聚合状态 |
| 单视频状态 | 将请求结果拆成 `pending` / `ready` / `empty` / `unavailable` |
| 设置 UI | 功能分组 + 风险标签 + 状态一行 |
| 可选 | WBI 签名、从卡片 DOM 补时长/播放量等降级 |

## 待讨论问题

1. **默认开关**：标签 + 分区默认开启且走 API — 是否改为默认关，或在首次启用时弹简短说明？
2. **失败阈值**：连续几次失败标「不可用」？（建议 3–5 次）
3. **无数据时的语义**：是否需要在叠加层/debug 里显示「因 API 不可用，标签规则未执行」？
4. **论坛发布口径**：README 与帖子是否单独一节「能力边界」，与原版 Greasy Fork 页如何区分？
5. **WBI 优先级**：先做透明化，还是透明化与 WBI 并行？
